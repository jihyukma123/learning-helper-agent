# Action-Trigger Skill 사용 예시

이 문서는 `action-trigger` 스킬이 실제로 어떻게 작동하는지 보여주는 샘플 출력입니다.

## 사용자 입력 예시

> "오늘 '기술이 아닌 문제에 집중해야 함'이라는 걸 깨달았어. 근데 막상 일할 때는 또 기술부터 생각하게 되더라. 이걸 어떻게 실제로 적용할 수 있을까?"

## Action-Trigger 스킬 출력 예시

```
╔══════════════════════════════════════════════════════════════╗
║  🎯 Action Trigger: 문제 정의가 먼저다                          ║
╚══════════════════════════════════════════════════════════════╝

────────────────────────────────────────────────────────────────
💡 깨달음 재구성
────────────────────────────────────────────────────────────────

핵심 원칙:
"솔루션/기술을 선택하기 전에, 해결하려는 문제를 명확히 정의해야 한다"

왜 중요한가:
비즈니스 문제를 해결하는 엔지니어가 되려면, 기술 선택은 문제가 
명확해진 후의 일입니다. 잘못된 문제를 해결하는 것보다 비효율적인 
일은 없습니다.

해결하려는 문제:
• 무의식적으로 "어떤 기술을 쓸까?"부터 생각하는 습관
• 요구사항을 받자마자 구현 방법부터 고민
• 문제의 본질을 놓치고 표면적 요청만 충족하는 경우

────────────────────────────────────────────────────────────────
🎭 체화 차원 분석
────────────────────────────────────────────────────────────────

▸ A. 반응적 체화 (추천 집중 영역)

당신의 경우, 이 차원에서 시작하는 것을 강력히 권장합니다.

트리거 상황:
• 회사에서 "이 기능 만들어주세요" 요청 받을 때
• PM이나 동료가 개선 요청할 때
• 팀 회의에서 기술 스택 논의가 시작될 때

이 상황들에서 자동으로 "잠깐, 문제가 뭐지?"라고 물어보는 것이 
첫 번째 목표입니다.

▸ B. 습관적 체화

두 번째 단계로, 새로운 작업을 시작할 때의 루틴에 통합:
• 새 사이드 프로젝트 시작 시
• 리팩토링 시작 전
• 새 라이브러리 도입 검토 시

코드 에디터를 열기 전에 반드시 "문제 정의 문서" 작성을 거치는 
습관을 만드는 것이 목표입니다.

▸ C. 사고방식 체화 (장기 목표)

궁극적으로는 모든 의사결정의 기저에 "Problem-First Thinking"이 
깔리는 것이 목표입니다. 하지만 이는 A, B가 충분히 체화된 후에 
자연스럽게 확장됩니다.

────────────────────────────────────────────────────────────────
⚡ 구체적 행동 트리거 설계
────────────────────────────────────────────────────────────────

트리거 #1: 업무 요청 받을 때

📍 상황 (Situation)
  - 언제: 슬랙/이메일로 기능 요청 메시지 받을 때
  - 어디서: 회사 업무 중
  - 트리거 신호: "~를 만들어주세요", "~를 추가해주세요"

⚡ 행동 (Action)
  1. 즉시 답장하지 않기 (5분 대기)
  2. 다음 3가지 질문 중 1개 이상 하기:
     • "이게 해결하려는 문제가 정확히 뭔가요?"
     • "사용자가 겪고 있는 불편함이 뭔가요?"
     • "이게 해결되면 어떤 결과를 기대하시나요?"
  3. 답변을 노트에 기록
  
  소요 시간: 5-10분
  도구: 슬랙, 노션 페이지

📊 측정 (Measurement)
  - 추적 방법: 금요일 30분, 이번 주 슬랙 메시지 리뷰
  - 성공 기준: 80% 이상의 요청에 대해 문제 파악 질문
  - 기록 양식: "이번 주 기능 요청: X건, 문제 파악 질문: Y건"

---

트리거 #2: 새 프로젝트 시작할 때

📍 상황 (Situation)
  - 언제: "이거 만들고 싶다" 생각이 들 때
  - 어디서: 개인 개발 시간
  - 트리거 신호: 코드 에디터를 열려고 하는 순간

⚡ 행동 (Action)
  에디터 열기 전 필수 절차:
  1. 프로젝트 폴더 생성
  2. problem-definition.md 파일 생성
  3. 템플릿 채우기:
     • 해결하려는 문제: [1-3문장]
     • 왜 이게 문제인가: [근거]
     • 성공 기준: [측정 가능한 지표]
  
  소요 시간: 10-15분
  도구: 텍스트 에디터

📊 측정 (Measurement)
  - 추적 방법: 프로젝트 폴더에 problem-definition.md 존재 확인
  - 성공 기준: 모든 새 프로젝트에 파일 존재
  - 추적 주기: 월말 회고 시 확인

────────────────────────────────────────────────────────────────
🗺️ 체화 로드맵
────────────────────────────────────────────────────────────────

Phase 1: 의식적 적용 (1-2주차)
  목표: 트리거 인식 & 의도적 행동 시작
  
  • Week 1: 트리거 #1만 집중
    - 매일 아침: "오늘 요청 받으면 질문하기" 리마인더
    - 매일 저녁: 체크리스트 확인
    - 성공 기준: 30% 이상 실행
  
  • Week 2: 트리거 #1 계속 + 트리거 #2 추가
    - 두 가지 트리거 모두 의식적으로 기억하기
    - 성공 기준: #1은 50%, #2는 30%

Phase 2: 반복 강화 (3-4주차)
  목표: 실행률 60% 이상, 자연스러움 증가
  
  • 리마인더 의존도 줄이기
  • 주간 회고에서 패턴 분석:
    - 어떤 상황에서 잘 실행됐나?
    - 어떤 상황에서 놓쳤나?
  • 성공 기준: 평균 60% 실행률

Phase 3: 자동화 (5주차 이상)
  목표: 의식적 노력 없이 자동 실행
  
  • 트리거 발생 시 자동으로 "잠깐, 문제가 뭐지?" 생각 떠오름
  • 질문/문서 작성이 귀찮지 않고 자연스러움
  • 성공 기준: 90% 실행률
  • 체화 확인: 동료가 "너는 항상 문제부터 묻더라" 피드백

────────────────────────────────────────────────────────────────
🔔 리마인더 시스템
────────────────────────────────────────────────────────────────

환경 단서:
  • 모니터 하단에 "문제가 뭐지?" 포스트잇 부착
  • 노트북 상단에 "Problem First" 스티커
  • VS Code 워크스페이스에 problem-definition.md 템플릿 배치

디지털 리마인더:
  • 오전 9시: "오늘 업무 요청 받으면 문제부터 파악하기"
  • 점심 시간: "오전에 문제 정의했나 체크"
  • 슬랙 프로필 상태: "문제 정의 먼저 🎯"

사람 활용:
  • 가까운 동료에게: "나 요즘 문제 정의 먼저 하는 습관 들이는 중인데,
    내가 바로 솔루션 얘기하면 '문제가 뭔데?'라고 물어봐줘"

프로세스 통합:
  • 커피 타는 시간 → 받은 요청 검토 & 질문 준비
  • 출근 후 첫 10분 → 어제 요청 문제 정의 확인
  • 금요일 회고 → 이번 주 실행률 체크

## 스킬 활성화 방법

AI Agent에게 다음과 같이 요청하세요:

```
"[깨달음 내용]을 실제로 어떻게 적용할 수 있을까?"
"이 인사이트를 습관으로 만들고 싶어"
"이걸 체화하려면 어떻게 해야 해?"
```
